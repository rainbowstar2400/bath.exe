# Bath.exe 〜いいから入れ〜 仕様書

## 1. 概要

### 1.1 コンセプト

「入りたいのに入れない」を解決するPWAアプリ。
定時にプッシュ通知で煽り、チャットで駄々をこねても最終的にお風呂に送り込む。
「今はいる！」で後押しし、「はいった！」で褒める。

### 1.2 技術スタック

| レイヤー | 技術 |
|---|---|
| フロント | PWA（HTML/JS）+ Service Worker |
| バックエンド | Vercel サーバーレス関数 |
| プッシュ通知 | Web Push API + `web-push`（VAPID認証） |
| AI | Claude API（claude-sonnet-4-20250514） |
| DB | Supabase |
| 定時実行 | Supabase pg_cron + pg_net |
| 認証 | Supabase 匿名認証 |

## 2. 機能仕様

### 2.1 プッシュ通知（段階的煽り）

設定時刻を起点に3回、段階的にプッシュ通知を送信する。
デフォルトの通知開始時刻は **23:00**。

| 段階 | タイミング | 強度 | タイトル例 | 本文例 |
|---|---|---|---|---|
| 1回目 | 設定時刻 +0分 | やさしめ |
| 2回目 | 設定時刻 +15分 | 煽り |
| 3回目 | 設定時刻 +30分 | 圧 |

- **通知文は Claude API で毎回動的生成する**（段階と強度を指定し、毎日異なる煽り文を生成）
- 通知にはバイブレーションパターンを設定する（`vibrate` オプション）
- 「今はいる！」フェーズでは見積もりメッセージを更新する（2.4参照）
- 段階が進むほどバイブレーションを長くする
- 通知タップでアプリのチャット画面を開く
- 送信前に「入った！」フラグをチェックし、済みならスキップする

### 2.2 チャット（駄々こね機能）

Claude APIを使い、ユーザーの言い訳に対応するチャット機能。

**システムプロンプト方針：**
- 共感はするけど絶対に折れない
- 図星をついてくるトーン
- どんな言い訳にも切り返す
- 最終的には必ず「入るよ！」に着地させる

**セッション管理：**
- 会話履歴は `chat_sessions` テーブルに保存
- 毎日 12:00（正午）にセッションをリセット（新セッション開始）
- Claude API にはセッション内の全会話履歴を送信し、文脈を維持する

**メッセージ上限：**
- 1セッションあたり **10往復** まで
- 上限に達したらチャット入力を無効化し、「もう言い訳は聞かない。入れ。」のようなメッセージを表示
- 画面上にも残り回数を表示する（例：「残り 7/10」）

### 2.3 時計表示と入浴タイマー

メイン画面上部（ナビの下）に現在時刻と入浴見積もりメッセージを表示する。

**現在時刻：**
- 大きめのフォントで常時表示（毎秒更新、秒は小さめに表示）

**見積もりメッセージ（小さめに表示）：**

| フェーズ | メッセージ |
|---|---|
| 初期状態 | 今入れば XX:XX には出れるよ！ |
| 「今はいる！」後 | XX:XX くらいには出れるね！ |
| 「はいった！」後 | 今日は XX:XX に入れました！ |

- 見積もり時刻 = 現在時刻 + お風呂の時間（設定値）
- 「はいった！」後の時刻は実際に「はいった！」を押した時刻
- お風呂の時間は設定画面で変更可能（3.2参照）、localStorage に保存

### 2.4 入浴ボタン（2段階）

メイン画面下部に大きく配置する入浴アクションボタン。2段階の操作フローで入浴を促す。

**フェーズ1：「今はいる！」**
- タップすると、チャットにユーザー吹き出し「今はいる！」を表示
- AIが後押しメッセージを返す（例：「いいぞいいぞ！いってらっしゃい！」）
- `POST /api/bath/start` で入浴開始時刻（`started_at`）を記録
- ボタンが「はいった！」に変化（ブルー → コーラルに色変化）

**フェーズ2：「はいった！」**
- お風呂から出た後にタップ
- チャットにユーザー吹き出し「はいった！」を表示
- `POST /api/bath/done` で記録 + Claude API で褒めメッセージを生成
- ボタンが「おつかれ！」に変化（グリーンに色変化、操作不可）
- 未送信の残り通知をキャンセルする

**褒め方のルール：**
- 設定時刻より前に押した → 激褒め（例：「え、早くない！？天才！？」）`praise_level: excellent`
- それ以外 → 普通に褒める（例：「えらい！おつかれ！」）`praise_level: good`

**チャット表示の区別：**
- ボタンアクションによる送信/返答はブルー系の色で表示
- 通常の言い訳チャットはピーチ（ユーザー）/ 白（AI）で表示
- 各メッセージに送信時刻を表示

### 2.5 初回起動オンボーディング

初回起動時にポップアップ形式で基本設定と使い方を案内する。
初回起動の判定は localStorage のフラグ（`onboarding_done`）で管理する。

**ステップ1：ようこそ**
- 「ようこそ！　Bath.exe は、あなたがお風呂に入るのを後押しするアプリです」
- [つぎへ] ボタン

**ステップ2：通知時刻**
- 「何時にお風呂に入りたい？」
- タイムピッカー（デフォルト 23:00）
- 「この時間に後押しの通知が届きます！」
- [つぎへ] ボタン

**ステップ3：お風呂の時間**
- 「お風呂にかかる時間はどのくらい？」
- セレクトボックス（5分刻み、5〜90分、デフォルト20分）
- 「『今入ったらいつ出れるか』がわかります！」
- [つぎへ] ボタン

**ステップ4：通知許可**
- 「通知を許可してね！」
- 「後押しの通知を届けるために必要です」
- [通知を許可する] ボタン
- [あとで] リンク（スキップ可能）

**ステップ5：完了**
- 「準備完了！」
- 「言い訳があれば聞きますが、最終的には入ってもらいますよ」
- [はじめる] ボタン

**保存タイミング：**
- ステップ2の通知時刻とステップ3のお風呂の時間は、完了時にまとめて保存
- 通知時刻はサーバー（`POST /api/settings/update`）、お風呂の時間は localStorage
- ステップ4の通知許可はブラウザの許可ダイアログ経由で即時実行

### 2.6 記録画面

過去7日間（今日を含むスライディングウィンドウ）の入浴記録を表示する。

**表示内容：**
- 連続記録（小さめに表示、重要度は低い）
- 過去7日の入浴日数（分数表示：例「5/7日」）
- 日別カード（各日の入浴記録をリッチに表示）

**日別カードの表示項目：**
- 日付（○月○日・曜日）
- 入った時刻（`started_at`）— メイン情報
- 出た時刻（`done_at`）— サブ情報
- 所要時間（`done_at - started_at` の差分）
- ⭐早い！マーク（`praise_level: excellent` の場合のみ）

**カードの状態パターン：**

| 状態 | 条件 | 表示 |
|---|---|---|
| 入浴済み（両方記録あり） | `started_at` + `done_at` あり | 通常表示 |
| 入ったが出た記録なし | `started_at` のみ | `done_at` は設定の所要時間から推測、「推測値」マーク付き |
| 未入浴 | 記録なし（アプリ使用開始後） | ❌ 表示 |
| アプリ使用開始前 | アプリ登録日より前の日付 | グレーアウト（❌ とは区別） |

**編集機能：**
- カードをタップすると入った時刻・出た時刻を編集可能
- 過去の日にも手動で記録を追加可能（「とっくに入ってた」ケース）
- 編集は `POST /api/bath/edit` で保存

## 4. API 仕様

### 4.1 `POST /api/subscriptions/register`

サブスクリプション情報を登録・更新する。

**リクエスト：**
```json
{
  "endpoint": "https://fcm.googleapis.com/...",
  "keys": {
    "auth": "...",
    "p256dh": "..."
  }
}
```

**処理：**
1. Supabase 匿名認証のトークンからユーザーIDを取得
2. `subscriptions` テーブルに upsert

### 4.2 `POST /api/push/send`

pg_cron から叩かれ、対象ユーザーにプッシュ通知を送信する。

**リクエスト：**
```json
{
  "stage": 1
}
```

**処理：**
1. `enabled = true` かつ現在時刻が通知対象のユーザーを `subscriptions` から取得
2. 当日の「入った！」フラグが立っていたらスキップ
3. 段階（stage）に応じた通知文を Claude API で動的生成
4. `web-push` ライブラリで送信
5. 送信失敗で 410 が返った場合、該当サブスクリプションを DB から削除（期限切れ処理）

**認証：** pg_cron からの呼び出しのため、シークレットキーで認証する。

### 4.3 `POST /api/chat`

チャットメッセージを処理する。

**リクエスト：**
```json
{
  "message": "あと5分だけ..."
}
```

**処理：**
1. ユーザーIDから当日のセッションを取得（なければ新規作成）
2. 会話履歴にユーザーメッセージを追加
3. システムプロンプト＋会話履歴を Claude API に送信
4. レスポンスを会話履歴に追加して保存
5. レスポンスを返却

**レスポンス：**
```json
{
  "reply": "5分後って言って5分で済んだことある？ほら、立って。"
}
```

### 4.4 `POST /api/bath/start`

「今はいる！」を処理し、入浴開始を記録する。

**処理：**
1. `bath_logs` に新規レコードを作成（`started_at` = 現在時刻、`done_at` = null）
2. 作成したレコードの ID を返却（後で `bath/done` で更新するため）

**レスポンス：**
```json
{
  "log_id": "uuid",
  "started_at": "2026-03-28T22:48:00+09:00"
}
```

### 4.5 `POST /api/bath/done`

「はいった！」を処理する。

**処理：**
1. 当日の `bath_logs` レコード（`done_at` が null のもの）を取得
2. `done_at` = 現在時刻で更新
3. 現在時刻と設定時刻を比較して `praise_level` を判定（excellent / good）
4. 当日の残り通知をキャンセル（送信済みフラグを立てる）
5. Claude API で褒めメッセージを生成
6. レスポンスを返却

**レスポンス：**
```json
{
  "reply": "え、まだ22時50分！？早すぎ！天才！偉すぎる！！！",
  "praise_level": "excellent"
}
```

### 4.6 `POST /api/bath/edit`

入浴記録を手動で編集・追加する。

**リクエスト：**
```json
{
  "date": "2026-03-27",
  "started_at": "2026-03-27T23:00:00+09:00",
  "done_at": "2026-03-27T23:25:00+09:00"
}
```

**処理：**
1. 指定日付の `bath_logs` レコードを取得（なければ新規作成）
2. `started_at`、`done_at` を更新
3. `praise_level` を再計算（設定時刻との比較）
4. `is_manual: true` フラグを設定（手動編集であることを記録）

**レスポンス：**
```json
{
  "ok": true
}
```

### 4.7 `GET /api/stats`

記録画面用のデータを取得する。

**レスポンス：**
```json
{
  "streak": 3,
  "registered_at": "2026-03-23T10:00:00+09:00",
  "days": [
    {
      "date": "2026-03-28",
      "started_at": "2026-03-28T22:48:00+09:00",
      "done_at": "2026-03-28T23:10:00+09:00",
      "praise_level": "excellent",
      "is_manual": false
    },
    {
      "date": "2026-03-27",
      "started_at": "2026-03-27T23:12:00+09:00",
      "done_at": "2026-03-27T23:35:00+09:00",
      "praise_level": "good",
      "is_manual": false
    },
    {
      "date": "2026-03-26",
      "started_at": null,
      "done_at": null,
      "praise_level": null,
      "is_manual": false
    },
    {
      "date": "2026-03-25",
      "started_at": "2026-03-25T23:31:00+09:00",
      "done_at": null,
      "praise_level": "good",
      "is_manual": false
    }
  ]
}
```

**処理：**
1. `bath_logs` から過去7日分（今日〜7日前）のデータを取得
2. 連続記録（streak）を算出
3. ユーザーの登録日（`registered_at`）を返却（グレーアウト判定用）
4. `done_at` が null で `started_at` がある場合、フロント側で設定の所要時間から推測表示

### 4.8 `POST /api/settings/update`

通知設定を更新する。

**リクエスト：**
```json
{
  "notify_time": "22:30",
  "enabled": true
}
```

**処理：**
1. `subscriptions` テーブルの該当レコードを更新
2. pg_cron のジョブを `cron.unschedule` → `cron.schedule` で再登録

## 5. DB スキーマ

### 5.1 subscriptions

| カラム | 型 | 制約 | 説明 |
|---|---|---|---|
| id | uuid | PK, default gen_random_uuid() | |
| user_id | uuid | NOT NULL, UNIQUE | Supabase auth のユーザーID |
| endpoint | text | NOT NULL | プッシュエンドポイントURL |
| auth | text | NOT NULL | VAPID auth キー |
| p256dh | text | NOT NULL | VAPID p256dh キー |
| notify_time | time | NOT NULL, default '23:00' | 通知開始時刻 |
| enabled | boolean | NOT NULL, default true | 通知オンオフ |
| created_at | timestamptz | default now() | 作成日時 |
| updated_at | timestamptz | default now() | 更新日時 |

### 5.2 chat_sessions

| カラム | 型 | 制約 | 説明 |
|---|---|---|---|
| id | uuid | PK, default gen_random_uuid() | |
| user_id | uuid | NOT NULL | Supabase auth のユーザーID |
| messages | jsonb | NOT NULL, default '[]' | 会話履歴（Claude API形式） |
| session_date | date | NOT NULL | セッションの日付 |
| created_at | timestamptz | default now() | 作成日時 |

**ユニーク制約：** `(user_id, session_date)`

**セッション日付の判定：**
- 正午（12:00 JST）を日付の境界とする
- 例：3/27 12:00 〜 3/28 11:59 → session_date = '2025-03-27'

## 6. プッシュ通知フロー

### 6.1 初回登録

```
ユーザー → アプリ起動
         → 通知許可ダイアログ
         → 許可
         → Service Worker がサブスクリプション取得
         → POST /api/subscriptions/register
         → Supabase に保存
```

### 6.2 定時通知

```
pg_cron (23:00 JST)
  → pg_net で POST /api/push/send { stage: 1 }
  → Vercel 関数が Supabase から対象ユーザーを取得
  → web-push で通知送信
  → スマホに通知表示

pg_cron (23:15 JST)
  → POST /api/push/send { stage: 2 }
  →（同上）

pg_cron (23:30 JST)
  → POST /api/push/send { stage: 3 }
  →（同上）
```

### 6.3 「入った！」による通知キャンセル

```
ユーザー → 「入った！」ボタン
         → POST /api/bath/done
         → 当日の残り通知をスキップするフラグを立てる
         → /api/push/send は送信前にフラグをチェックし、スキップ
         → 褒めメッセージを返却
```
